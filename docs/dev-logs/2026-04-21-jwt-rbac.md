## 대상

- 이슈: `#26`

- 범위: Supabase JWT 검증, claim-to-role 매핑, 역할 기반 API 접근 제어

## A/B 비교

### A. 기존 개발용 헤더 인증 유지 + 컨트롤러/서비스에서 역할 분기 강화

- 장점: 변경 폭이 작고 빠르게 통과시킬 수 있다.
- 단점: 운영 신뢰 경계가 여전히 헤더 기반이라 이슈 목표를 충족하지 못한다.

### B. Spring Security resource server로 JWT Bearer 검증 전환

- 장점: issuer, audience, expiry를 서버 경계에서 검증할 수 있고 역할 매핑을 중앙화할 수 있다.
- 단점: 보안 설정, 테스트 토큰 생성, 기존 서비스 예외 흐름을 함께 정리해야 한다.

## 선택

- `B`를 선택했다.
- 이유: 이 이슈의 핵심은 운영용 인증 경로 준비이며, 헤더 인증 유지 방식은 수락 기준을 만족하지 못한다.

## 구현 요약

- `spring-boot-starter-oauth2-resource-server` 추가
- `app.security.jwt.*` 설정 추가
- Supabase JWT claim을 `PLANNER`, `FINANCE_REVIEWER`, `EXECUTIVE` 권한으로 변환
- 개발용 `HeaderAuthenticationFilter` 제거
- `WorkflowController`와 `ApprovalWorkflowService`의 역할 범위를 JWT/RBAC 기준으로 정리
- 401/403 보안 테스트 추가 및 서비스 테스트 정리
