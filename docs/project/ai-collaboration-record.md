# CostWiseAI AI 협업과정 기록

## 1) 협업 목적

AI 협업의 목적은 속도를 높이는 것이 아니라, 이슈 완료조건을 더 정확하게 충족하면서 변경 범위를 통제하는 것이다.

## 2) 운영 전략

### 전략 A: 이슈 기반 수직 슬라이스
- 한 번에 큰 리팩터링 대신 이슈 완료조건에 직접 대응하는 작은 수직 슬라이스를 먼저 구현
- 예: #49(근거 가시화 1차) → #51(평가 근거 구조 1차)

### 전략 B: 증거 중심 게이팅
- 구현 후 반드시 빌드/테스트 명령으로 증거 확보
- 증거 없이는 “완료” 판단 금지

### 전략 C: 문서 동시화
- 코드와 함께 `docs/dev-logs`에 변경 이유/검증/리스크를 기록
- 추후 회고/감사 대응 가능하도록 텍스트 증적 유지

## 3) 에이전트 역할 정의

### 3.1 관리자 에이전트 (Orchestrator)
- 모델: GPT-5.4
- 책임:
  - 이슈 우선순위 결정
  - 작업 단위 분해
  - 검증 게이트 통과 여부 판단
  - 커밋/PR/이슈 close 의사결정

### 3.2 작업 에이전트 (Worker)
- 모델: GPT-5.3-codex
- 책임:
  - 지정된 파일 범위 구현
  - 테스트/빌드 실행 및 결과 보고
  - 변경 파일 목록/잔여 리스크 명시

### 3.3 필요 시 탐색 에이전트 (Explorer)
- 모델: GPT-5.3-codex (또는 경량 탐색용 모델)
- 책임:
  - 코드베이스 질의 응답
  - 영향 범위 파악

## 4) 실제 협업 패턴

1. 관리자 에이전트가 이슈 완료조건을 재확인  
2. 작업 범위를 백엔드/프론트/문서로 분리  
3. 병렬 가능하면 Worker에게 파일 소유권을 분리해 배분  
4. 통합 후 관리자 에이전트가 테스트/빌드 재검증  
5. 커밋/PR 생성 및 이슈 상태(유지/종료) 판단

## 5) 모델 선택 기준

- GPT-5.4(관리자):
  - 이슈 해석, 우선순위, 트레이드오프, 종료 기준 판단에 사용
- GPT-5.3-codex(작업):
  - 구현/수정/테스트 반복이 많은 코드 작업에 사용

## 6) 기록 규칙

- 모든 기능 슬라이스는 `docs/dev-logs/YYYY-MM-DD-*.md` 생성
- PR 본문에는 최소 항목 유지:
  - 요약 / 변경 내용 / 이유 / 검증 / 체크리스트
- 이슈 종료는 “완료조건 충족”이 확인된 경우에만 수행

## 7) 리스크와 완화

- 리스크: 모델이 완료조건을 과대해석해 이슈를 조기 종료할 수 있음
  - 완화: 이슈 본문의 완료조건을 체크리스트로 재작성 후 검토
- 리스크: 병렬 작업 시 파일 충돌/맥락 누락
  - 완화: 파일 소유권을 명시하고, 통합 시 관리자 단일 판단
- 리스크: 검증 누락
  - 완화: backend test + frontend lint/build를 기본 게이트로 강제
